# 持久工作流状态规范

仅用于 Deep 或持续时间长、跨多个阶段的 Standard 任务。Quick 任务不得创建 `workflow_status.md`；短 Standard 任务优先使用当前会话中的简洁计划。

如果项目已有任务看板、规范目录或等价状态文件，复用既有位置和格式，不重复创建。状态文件记录可恢复的阶段事实，不记录逐条工具调用和隐藏思维过程。

## 一、更新时机

在以下事件后更新状态：

- 完成初始侦察和需求契约；
- 计划或依赖关系发生变化；
- 一个节点开始、通过、失效或等待外部条件；
- 获得会改变根因、范围或风险的新证据；
- 完成一轮验证、Critic 审查或修复复验；
- 出现需要用户授权或外部状态的阻塞；
- 最终验收完成。

不要仅因读取文件或执行普通命令而更新。

## 二、状态枚举

节点状态只使用：

- `pending`：依赖未满足或尚未开始；
- `in_progress`：当前正在处理，任一时刻最多一个主节点；
- `needs_fix`：实现或验证发现必须修复的问题；
- `waiting_external`：需要不可替代的用户决定、凭证、权限或外部状态；
- `passed`：验收标准满足且存在新鲜证据；
- `superseded`：因新证据或计划调整被明确替代。

没有新鲜验证证据的节点不得标记为 `passed`。

## 四、状态转换规则

- 节点只有在依赖均为 `passed` 或被明确替代后才能进入 `in_progress`。
- 验证失败时进入 `needs_fix`，修复后重新进入验证，不直接转为 `passed`。
- 生产代码、测试、共享接口、Schema、状态机或环境变化使相关旧证据失效时，将受影响节点从 `passed` 改回 `pending` 或 `needs_fix`。
- `waiting_external` 必须写明最小缺失条件；仍可执行的安全工作继续推进。
- `superseded` 必须记录替代它的新节点或决策，不能用来隐藏未完成工作。
- 最终完成要求所有 MUST 对应节点为 `passed`，P0/P1 为零，审查和验证证据保持新鲜。

## 五、证据引用与精简

- 状态文件引用测试输出、差异、日志或报告的位置，不复制大量原始输出。
- 对密钥、令牌、隐私数据和敏感路径进行脱敏。
- 每轮只保留会影响后续工作的结论，过期结论标记失效或被替代。
- 状态文件不能替代真实测试、代码审查或运行验证。
- 任务完成后保留最终结论和必要证据索引，删除无价值的过程噪声。
