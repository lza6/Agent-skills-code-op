# 老板式结果对齐协议

这份参考用于把模糊的“帮我做一下”转换为可交付、可验收的结果，同时不把技术细节决策推给用户。

## 一、先找到“最后看到什么”

从当前上下文提取：

1. 谁在什么入口开始；
2. 他要完成什么操作；
3. 成功后屏幕、API、数据、文件或运行状态最终呈现什么；
4. 失败、空数据、超时、重复操作、权限不足时看到什么；
5. 什么证据足以证明不是“写了代码”，而是“结果已经实现”。

把结果写成可观察的状态，不要写“优化一下”、“提高质量”、“使用最佳实践”这类无法验收的话。

## 二、什么时候必须问

只有当不同答案会实质改变以下任一项时才问：

- 用户可见行为或产品边界；
- 数据模型、公共接口或兼容性；
- 安全、隐私、权限或多租户隔离；
- 真实费用、外部写入、生产操作或不可逆行为；
- 交付层级：只要方案、只要审查、可运行原型，还是完整生产级落地；
- 会导致大量工作作废的关键业务假设。

库、文件名、内部分层、普通命名和可逆的实现细节，由智能体根据仓库事实决定。

## 三、选项的写法

每个问题给两到三个互斥选项，推荐项放第一个。选项标题尽量是“最终交付形态”，描述说明产物、边界和代价。

示例：

### 交付深度

- **直接生产落地（推荐）**：完成实现、测试、失败路径、文档和可交付验证，不执行需要批准的生产或外部操作。
- **先审查再决定**：只输出现状、根因、风险和分级修复清单，不修改代码。
- **先做方案或原型**：提供规格、架构或可演示原型，不宣称已达到生产标准。

### 兼容性取舍

- **保持旧行为（推荐）**：新能力以向后兼容方式加入，可能保留一层过渡代码。
- **明确破坏性升级**：删除旧行为并同步迁移调用方，必须有清晰的迁移和回滚方案。

### 产品行为

- **自动完成（推荐）**：用户提交后由系统完成全流程，只在异常或风险点需要人工介入。
- **每次确认**：关键步骤前都由用户确认，更可控，但操作成本更高。
- **只给建议**：系统不执行动作，只生成建议、草稿或可审核产物。

不要机械地复用上述选项，必须改写成当前任务的真实结果。

## 四、对齐后的内部任务契约

```md
目标：
[这次最终要实现什么]

用户旅程：
[用户从哪里进入，做什么，最后看到什么]

MUST 验收标准：
- [可观察标准 1]
- [可观察标准 2]

范围：
- 允许修改：
- 必须保留：
- 明确不做：

验证证据：
- [命令、测试、页面、API、数据或其他证据]

审批边界：
- [外部写入、生产、付费、破坏性操作]

停止条件：
- [什么状态算完成]
- [什么阻塞必须交回用户]
```

这是智能体内部的执行契约，不是每次都要把空表发给用户填写。
