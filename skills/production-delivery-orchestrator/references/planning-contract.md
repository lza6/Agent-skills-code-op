# 自适应计划契约

仅在任务需要制定实施计划时读取本文件。计划服务于交付，不得代替实现，也不得让用户决定可由仓库事实推导的普通技术细节。

## 一、先建立事实基础

计划前先确认最小充分事实：

- 用户要获得的可观察结果；
- 当前请求是分析、设计，还是已经授权修改；
- 适用的项目规则和未提交改动；
- 相关入口、模块、接口、数据流、状态流和用户旅程；
- 已有实现模式、测试入口、构建方式和可复用能力；
- 会改变结果、兼容性、数据、安全、成本或可逆性的未知项。

仓库能够回答的问题先自行调查。只有无法从代码、配置、测试、文档和安全诊断中确定的重要产品取舍，才询问用户。

## 二、选择计划深度

根据影响半径、风险和依赖关系选择最轻的充分计划。

### Quick

适用于局部、低风险、容易回滚且验收方式明确的任务。

计划使用一到三步，至少说明：

- 修改目标；
- 预计涉及的文件或模块；
- 最小验证方式。

不要创建流程文档、需求矩阵或大范围审计。

### Standard

适用于多文件功能、普通缺陷、接口或状态行为变化、中等影响重构。

计划至少包含：

- 目标和明确不做的内容；
- 涉及的文件、模块及其职责；
- 上下游依赖和执行顺序；
- 用户可观察的验收标准；
- 定向测试、接缝验证和必要烟雾测试；
- 兼容性、数据和回滚风险。

将任务拆成小批次，每一批都应能够独立验证，并尽量能够独立回滚。

### Deep

适用于跨模块或跨服务、公共契约、数据迁移、权限、计费、多租户、并发一致性、生产事故或高成本外部能力。

在 Standard 内容基础上补充：

- 需求与验收标准的追踪关系；
- 架构、数据流、状态流和失败恢复路径；
- 公共接口、数据模型、迁移与兼容策略；
- 并行节点、串行依赖和责任边界；
- 风险分级、最小证伪实验和停止条件；
- 分阶段发布、回滚和观测方案；
- 独立审查与最终复验安排。

只创建当前任务真正需要的工件，不机械生成空模板。

## 三、计划项的最低字段

每个实施节点都要能够回答：

```text
目标：该节点完成后有什么真实变化
范围：允许修改的文件或模块
依赖：开始前必须满足的条件和前置节点
验收：用户或调用方能观察到什么
验证：用什么测试、命令、运行结果或人工检查证明
回滚：失败时如何恢复，哪些旧行为必须保留
状态：pending / in_progress / blocked / passed
```

不要写“完善功能”“优化代码”“补充测试”这类无法验收的节点。把它改写为具体行为、文件范围和通过条件。

## 四、依赖与并行规则

- 有共享接口、Schema、迁移或同一文件写入时，优先串行。
- 文件范围互不重叠、输入输出稳定且合并方式清晰时，才并行。
- 调查、方案比较和只读审查可以并行，但必须由主执行者统一核对事实。
- 后续节点依赖的契约先确定并验证，再开始下游实现。
- 新证据推翻计划假设时，更新受影响节点，不继续执行已经失效的计划。

## 五、完成与阻塞

计划完成不等于任务完成。只有相关实现、验证和审查达到验收标准，节点才能标记为 `passed`。

以下情况才标记 `blocked`：

- 缺少不可替代的账号、凭证、权限或外部状态；
- 关键业务规则存在无法保守推断的冲突；
- 下一步需要生产写入、真实费用、不可逆操作或实质性扩大范围；
- 当前环境无法执行不可替代的验证。

报告阻塞时给出已完成内容、证据、最小缺失条件和解除阻塞后的下一步。
