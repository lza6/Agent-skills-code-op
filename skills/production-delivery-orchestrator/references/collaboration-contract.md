# 协作与安全合并契约

仅在任务明确需要多 Agent、并行侦察、分工修改或合并多个结果时读取。目标是提高独立工作吞吐，而不是让多个执行者并发写同一份状态。

## 一、父线程是唯一共享状态写入者

父线程负责：

- 划分任务、确认依赖、分配文件范围和维护验收标准；
- 写入共享状态、计划、汇总报告、发布记录、公共 changelog 与最终文档；
- 串行检查每个子任务的 diff，处理冲突，执行跨模块验证并宣布结果。

子任务不得并发修改上述共享状态，也不得替父线程合并其他子任务结果、推送、发布或最终签收。

## 二、先给每项工作分类

| 类型 | 是否可并行 | 写入规则 |
|---|---|---|
| 只读侦察、代码审查、日志/文档分析 | 可以 | 不写工作区；返回证据、涉及路径和不确定项 |
| 不重叠文件范围的实现或测试 | 可以 | 每项工作拥有明确文件范围；不得修改范围外文件、共享配置、迁移、锁文件或汇总文档 |
| 同一文件、公共契约、共享数据、生成制品、迁移、版本/发布元数据 | 不可以 | 由一个所有者串行执行；其他 Agent 只读提供建议或证据 |

不确定范围是否重叠时，默认串行。不要以“多人都很快”为由绕过共享状态保护。

## 三、给子任务一份可合并的工作契约

父线程在派发前明确：

```text
任务：要回答或实现的具体结果
输入：基线提交、相关需求和已知约束
允许写入：精确文件或互不重叠的目录范围；只读任务写“无”
禁止写入：共享状态、公共元数据、其他任务范围和用户已有无关改动
交付：修改路径、验证命令及结果、未解决风险或冲突
```

子任务完成时只报告事实；父线程再将结果合并到共享状态。失败或发现冲突时停止该子任务的写入，让父线程重新划分或转为串行，不要自行覆盖邻近结果。

## 四、稳定汇合而不制造父线程瓶颈

父线程不必串行完成所有工作：优先批量并行只读任务，或批量并行不重叠文件范围的任务。每一批结束后：

1. 收集每项任务的修改路径、验证结果与未解决项；
2. 串行核对 diff、依赖和共享边界；
3. 仅在无冲突且验收仍成立时合并下一批；
4. 对跨范围问题集中修复并重跑受影响验证。

这样父线程只承担协调和最终写入，不成为所有实现细节的瓶颈。

## 五、跨 CLI 与降级

不要假设任何 CLI 都支持并行子 Agent。没有该能力时，仍按同一所有权模型串行执行：先完成一项，再记录其证据和范围后继续下一项。不得把串行降级写成已经并行或独立合并。
