# 维护、终局审计与复用契约

适用于已有仓库、长期任务恢复、生产交付前审计、用户要求“继续”“全部闭环”“复用上次工作流”或要求先读取文档/规则/记忆再编码的任务。它补充而不替代 discovery、planning、execution、validation 和 review 契约。

## 1. 先读后改

在写代码、配置、测试或文档之前，按最小充分范围读取：

1. 仓库规则（`AGENTS.md`、`CLAUDE.md`、贡献指南）、入口 README 和当前状态文档；
2. 活动计划/规格/任务、`workflow_status.md`、最近审查和测试证据；
3. 与请求直接相关的源码、测试、配置与发布定义；
4. `git status --short`、当前分支/HEAD 和受影响文件的 `git diff`。

外部会话记忆只能用作线索，不能替代当前文件和命令证据。不要读取不相关的私有日志、凭证或用户数据；仓库外信息需要时只提取完成当前任务所必需的事实。

## 2. 文档事实新鲜度

把每一条“当前”“已通过”“已发布”“支持”或测试计数与当前代码/命令核对。至少检查：

- 文档所称版本、tag、commit、分支和发布日期是否仍匹配；
- 文档所称测试数、命令、环境、兼容矩阵和配置前提是否仍存在；
- 历史审计是否被清楚标为历史，且没有被当成 current-state；
- 当前 diff 是否使既有测试、截图、报告、覆盖率或发布证明失效。

将结果标为 `fresh`、`stale`、`partial` 或 `not_applicable`。事实不一致时先更新事实、补充回归或把状态降级；不要基于过时文档继续实现。没有对应架构面的项目必须明确写为“不适用”，不能伪造前端、数据库、API、容器或真实客户端验证。

## 3. 需求与执行节点

建立可恢复的简短矩阵：需求 → 位置 → 验收 → 新鲜证据 → 风险/回滚。Deep 或跨文件任务把节点写入 `workflow_status.md`，并记录依赖、唯一写入者、下一步和外部阻塞。

先修 P0/P1：启动/安装/核心调用、数据或文件安全、公开契约、认证/凭证、发布质量门。P2/P3 只在不扩大范围时处理；否则放入路线图。每个修改批次都要包含相称的异常、边界、兼容性、文档和验证工作，不用 TODO、mock 或静态命中文本代替真实链路。

## 4. 证据和验收

分别记录：

- **实现证据**：文件、接口/命令契约和兼容性影响；
- **运行证据**：实际执行的命令、结果、平台和时间；
- **调用证据**：安装、关键用户旅程或受影响失败路径；
- **限制**：不可替代的凭证、外部服务、付费调用或未执行环境；
- **回滚**：可逆文件批次、版本/tag 或明确不能自动回滚的外部操作。

不得把历史结果、静态评测、安装成功或 mock 当作当前运行证据。测试通过也不等于需求已完成；逐项映射 MUST 验收标准。外部 CLI、浏览器、模型和第三方服务只有实际调用并得到可保存的结果时才算真实运行。

## 5. 审查与结束

Deep、公开契约、安全边界、发布流程或用户明确要求多 Agent 时，安排独立只读 Critic。Critic 审查需求完整性、逻辑正确性、边界、代码质量、测试覆盖和实际运行结果；主线程修复确认问题后，让同一审查者复验受影响范围。

只有所有适用 MUST 有实现/验证映射、P0/P1 关闭、文档/配置/测试一致、未验证项诚实披露，才能标记该范围完成。缺凭证、权限或外部状态时，保留 `external_blocked`，给出最后一条可执行步骤，而不是把工作标为失败或完成。
