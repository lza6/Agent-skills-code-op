# 独立审查与复验契约

仅在高风险 Standard、Deep、用户明确要求独立审查，或实现涉及重要公共契约、安全、数据、计费和并发时读取本文件。

## 一、Builder 与 Critic 分权

### Builder

Builder 负责：

- 理解需求并维护验收标准；
- 实现当前范围内的改动；
- 执行初次验证；
- 提供当前差异、新鲜证据和已知限制；
- 修复 Critic 已确认的问题。

Builder 不得以自己的实现结论替代独立验收。

### Critic

Critic 只读，负责寻找能够被证据支持的缺陷。不得直接修改生产代码、测试、配置或审查对象，也不得替 Builder 完成修复。

Critic 至少获得：

- 原始需求和 MUST 验收标准；
- 当前相关代码或差异；
- 实际验证证据与未验证项；
- 适用的项目规则；
- 已知兼容性、数据和授权边界。

不要向 Critic 提供“当前实现为什么一定正确”的辩护性结论。允许提供设计决策，但必须同时提供约束和可反转条件。

## 二、审查范围

Critic 根据当前风险检查：

1. 需求是否遗漏或被偷换；
2. 正常逻辑和状态转换是否正确；
3. 错误、超时、取消、重复、并发和边界是否收敛；
4. 架构、类型、接口和数据是否与项目一致；
5. 测试是否能捕获真实回归，而不是只验证 Mock；
6. 运行证据是否新鲜、可追溯且覆盖核心旅程；
7. 安全、权限、租户、费用和敏感数据风险；
8. 性能、资源、重试和重复副作用；
9. 兼容性、迁移、回滚和文档一致性；
10. 当前差异是否混入范围外修改。

不要为了显得严格而制造问题。没有足够证据的担忧标记为“待验证”，不能写成已确认缺陷。

## 三、发现格式与结论

每个发现包含：

```text
严重程度：P0 / P1 / P2 / P3
关联需求：受影响的验收标准
位置：文件、模块或运行路径
证据：代码、测试、日志或复现方式
预期行为：正确结果
实际行为：当前结果
影响：用户、数据、兼容性或运行风险
修复方向：最小可行方向，不代替 Builder 编码
置信度：高 / 中 / 低
```

结论使用：

- `Request Changes`：存在已确认的 P0/P1；
- `Needs Discussion`：关键产品或业务取舍无法由证据决定；
- `Approve`：当前范围没有阻塞完成的问题，剩余项已披露。

P2/P3 不应为了追求“完美”无限阻塞当前交付，除非它会直接破坏 MUST。

## 四、修复与原 Critic 复验

Builder 根据问题清单修复后：

1. 重跑受影响验证，更新差异和证据；
2. 将修复结果交回同一个 Critic；
3. 原 Critic 复验原问题、相邻影响和新增回归；
4. 未关闭的问题保留原编号，补充新证据；
5. 新问题单独编号，不用模糊描述覆盖旧问题；
6. 直到 P0/P1 清零、MUST 满足，或出现真实外部阻塞才结束循环。

除非原 Critic 不再可用，否则不要用新的审查者替换复验者。必须更换时，向新 Critic 提供完整问题历史并披露连续性中断。

## 五、没有独立子执行者时的诚实降级

按以下顺序降级：

1. 优先使用独立上下文的只读审查者；
2. 如果只能创建独立会话或任务，传入最小必要原始材料进行只读审查；
3. 如果环境完全不支持独立上下文，由 Builder 暂停实现视角，按本契约执行隔离的第二遍审查。

第三种情况不是真正独立审查。最终报告必须写明：

- 未使用独立 Critic；
- 实际采用的是 Builder 自审降级；
- 自审覆盖的项目和仍可能存在的偏差；
- 哪些高风险结论仍需要外部复核。

不得把自审写成“独立 Critic 已通过”，也不得伪造审查线程、审查者或复验结果。
