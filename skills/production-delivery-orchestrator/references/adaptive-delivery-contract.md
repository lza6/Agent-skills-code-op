# 自适应交付强度契约

仅在需要快速建立代码库盘点、判断歧义是否值得提问、Deep 或高风险 Standard 的规格/质量审查，或选择核心逻辑的测试证据时读取本文件。目标是降低遗漏和定位成本，不把扫描器、两阶段审查或严格 TDD 机械施加到所有任务。

## 一、可选的代码库盘点

先用项目已有入口、文件列表、定向搜索、规则文件、配置和测试建立事实。需要给多栈仓库生成一个有边界的初始索引时，才可以运行：

```bash
python "<skill-dir>/scripts/repository_inventory.py" --root /path/to/repository
```

该脚本只读取受限数量的路径和文件名，输出相对路径、语言/构建信号、测试候选和入口候选；不会读取源码内容、执行项目代码或替代后续定向侦察。它是可选能力：Python 不可用、仓库过大、生成目录特殊或输出与实际证据冲突时，直接使用手工 `rg --files`、定向搜索和项目原有工具，不得把脚本或其输出当成完成门。

扫描后仍必须：

- 读取命中的真实入口、配置、调用链和测试，而不是只凭文件名推断；
- 排除依赖、构建产物和缓存，并在结果被截断时缩小范围或补充定向侦察；
- 将扫描发现与当前用户旅程、未提交改动和项目规则核对；
- 不将“扫描完成”写成“问题已定位”或“实现已验证”。

## 二、定性歧义路由

不要使用固定公式、加权总分、伪精确小数或阈值来决定是否提问。结合仓库证据，按以下定性风险维度判断：

| 维度 | 低 | 中 | 高 |
|---|---|---|---|
| 可见结果 | 现有行为和验收唯一 | 少量可兼容解释 | 不同选择会改变用户结果 |
| 兼容与数据 | 容易回滚且无共享数据 | 内部接口或状态变化 | 公共契约、迁移、权限、计费或租户边界 |
| 外部影响 | 本地、无费用 | 可替换集成 | 生产、真实费用、不可逆写入或合规影响 |
| 范围与证据 | 入口和根因清晰 | 需要有限侦察 | 多仓库、关键事实缺失或业务规则冲突 |

- **低**：采用可逆且符合现有模式的默认实现，在结果中披露假设。
- **中**：先补充仓库侦察、最小证伪实验或兼容性检查；证据足够时直接继续。
- **高**：只有无法保守推断、且不同答案会实质改变行为或授权边界时，带着已发现证据询问一到三个结果选项。

这不是风险排序算法；维度帮助解释路由理由，不能替代具体证据和产品决策。

## 三、两阶段规格/质量审查

仅对 **Deep** 或涉及公共契约、数据迁移、权限、计费、多租户、并发一致性、高影响用户旅程的 **高风险 Standard** 启用。Quick 和普通 Standard 不得因此新增空 Spec 或额外审查线程。

### 阶段 A：规格审查（实现前）

在编码前检查并留下最小可追踪结论：

- MUST、明确不做项、用户可观察的正常/失败/恢复结果；
- 接口、状态、数据、兼容性、授权和回滚边界；
- 关键未知项、定性风险、最小证伪实验和依赖顺序；
- 每项 MUST 的实现位置和验证方式。

有独立上下文时可让只读 Critic 审查这一结论；没有时执行与实现视角隔离的第二遍规格检查，并如实披露独立性下降。阶段 A 未解决的高风险歧义不得用实现猜测掩盖。

### 阶段 B：质量审查（实现与验证后）

将阶段 A 的 MUST 逐项对照当前 diff、运行证据、失败路径、兼容性和回滚。复用 `review-contract.md` 的 Builder/Critic 分权和复验规则；发现 P0/P1 后修复、重跑受影响验证并由原审查者复验。阶段 B 不得把测试名称、静态扫描或 Builder 自评当作质量结论。

## 四、核心逻辑的测试优先与等价证据

对可测试的核心业务逻辑、状态机、权限、计费、幂等、并发和缺陷复现，优先让测试或可重复诊断在生产改动前失败，再实施最小修复并使其通过。它的目的在于建立回归证据，不是追求测试仪式。

以下情况允许等价证据，但必须在验证记录中说明为什么不能先写自动化测试、修复前观察到什么、修复后如何复验：

- 遗留系统没有可用测试接缝，先使用最小输入、日志、合同调用或受控运行复现；
- 迁移、外部集成或并发问题需要 fixture、模拟服务、数据对账或集成测试；
- 文档、纯文本、构建元数据或无行为变化的任务，使用链接/渲染/构建/静态检查等对应证据。

不得以“遗留项目”为由跳过全部复现和回归证据，也不得只新增修复后的绿测试、放松断言、修改 Mock 或增加超时来伪造 TDD。
