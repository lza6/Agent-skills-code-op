# 生产级执行契约

仅在用户授权构建、修改、修复、重构或落地时读取本文件。执行应在当前仓库和当前任务范围内自主收敛，不把正常的工程步骤逐项交回用户批准。

## 一、确认本地授权边界

用户明确要求“修复”“实现”“完成”“改造”或等价结果时，视为已经授权：

- 读取和搜索相关代码、配置、测试与文档；
- 修改当前范围内的本地文件；
- 新增或更新相关测试；
- 运行非破坏性的测试、静态检查、构建和本地烟雾验证；
- 根据失败证据继续诊断、修复和复验；
- 更新与行为变化直接相关的文档。

只要求分析、审查、解释或规划时，不修改文件。

以下操作不属于默认本地授权，执行前必须获得明确批准：远程推送或合并、发布或部署、生产写入、真实外部写入、付费调用、破坏性数据操作、不可逆变更、系统级安装和实质性扩大范围。

## 二、保护工作区

修改前检查：

- 当前工作目录、仓库和分支；
- 工作区状态及未提交差异；
- 目标文件是否已有用户修改；
- 项目规则、生成文件边界和禁止修改区域。

把现有改动视为用户资产：

- 不重置、覆盖或清理无关改动；
- 不用破坏性命令换取“干净工作区”；
- 在已有目标文件修改上最小化合并；
- 无法安全区分用户改动与当前任务时，停止该文件写入并说明具体冲突；
- 修改后核对差异，只保留当前任务需要的变化。

## 三、先复现，再修复

对缺陷、新业务行为、状态机、权限、计费和可测试的核心逻辑，执行：

1. 用测试、诊断命令、最小样例或可观察运行结果复现现象。
2. 确认失败来自预期原因，而不是环境、错误测试假设或无关改动。
3. 优先添加能够捕获该行为的回归测试，并确认修复前失败。
4. 修改最小但完整的生产路径，覆盖直接上下游和必要失败状态。
5. 先运行定向验证，再运行受影响接缝和低成本关键路径验证。

如果无法合理先写自动化测试，记录等价复现和验证方式。不得因为测试基础不足而跳过所有证据，也不得只修改 Mock、放松断言或增加超时来冒充修复。

对遗留系统、外部集成、迁移、并发或文档/构建元数据等不适合严格测试先行的任务，读取 `adaptive-delivery-contract.md` 选择受控运行、fixture、对账、渲染或静态检查等等价证据。等价不表示更低标准：必须保留修复前观察、修复后复验和剩余限制。

## 四、小批次执行

- 每次只处理一批高相关变化，并在进入下一批前验证当前批次。
- 遵循项目现有架构、类型、命名、错误处理和测试模式。
- 优先复用现有能力，不为当前任务引入无必要依赖或无关重构。
- 完成真实正常路径、失败路径和边界路径，不留下当前范围内的占位、假返回或待办标记。
- 公共接口、共享数据、迁移或兼容行为发生变化时，先实现保护或过渡策略。

## 五、验证失败后的续修循环

当测试、静态检查、构建、渲染或烟雾验证失败时：

1. 判断失败属于实现缺陷、测试假设错误、环境缺失、依赖问题还是用户已有改动。
2. 收集能够区分候选原因的最小新证据。
3. 修复当前授权范围内的真实原因，不掩盖错误或降低有效门槛。
4. 根据影响半径重跑定向测试、接缝验证和必要烟雾测试。
5. 更新计划节点和验证证据；受新改动影响的已通过节点重新标记为待验证。
6. 只要下一轮安全、在范围内且能产生新证据或实质进展，就继续循环。

只有遇到不可替代的外部凭证、生产状态、真实费用、不可逆操作、关键业务决策或范围扩张时才停止。不得把仓库内可解决的测试失败直接交回用户。

## 六、有限的同模式缺陷扫描

当前问题验证通过后，搜索与根因相同的实现模式，但限制在：

- 当前模块及其直接调用方和被调用方；
- 共用同一接口、状态枚举、数据结构或基础封装的路径；
- 与当前用户旅程共享同一失败机制的代码；
- 能用现有证据和测试确认的相似实现。

直接修复同模式问题仅当它与当前任务紧密相关、根因一致、改动向后兼容且验证成本可控。其他发现记录为风险或后续项，不自动扩大为全库重写。

## 七、完成门槛

宣布完成前确认：

- 当前范围内的 MUST 验收标准都有实现和验证映射；
- 相关测试和必要工程检查已经实际执行；
- 关键用户旅程及受影响失败路径已经验证；
- 没有未解决的 P0/P1 问题；
- 工作区差异没有混入无关修改；
- 未验证项、外部限制、兼容性和回滚边界已经如实披露。

只报告实际观察到的结果。不得把计划、Mock、未执行命令或理论推断写成已完成。
