# 风险驱动验证契约

仅在任务需要验证实现、修复或行为变化时读取本文件。验证目标是用最小但充分的新鲜证据证明当前改动满足验收标准，不是机械运行所有测试。

## 一、建立验证映射

先把每项 MUST 验收标准映射到：

- 对应的实现文件或运行路径；
- 最可能失败的接缝和边界；
- 能直接证明行为的验证方式；
- 证据失效条件。

没有实现映射或验证方式的 MUST 不得标记完成。

## 二、按风险选择验证层级

### L1：局部低风险

适用于文案、局部样式、明确配置和容易回滚的机械修改。

通常选择：

- 定向测试或等价静态检查；
- 必要的构建、渲染或局部人工检查；
- 修改文件的差异核对。

### L2：中等影响

适用于多文件行为、组件或服务变化、内部接口和普通缺陷。

通常选择：

- 能复现目标行为的单元或组件测试；
- 受影响接缝的集成或合约验证；
- 一条核心用户路径的低成本烟雾验证；
- 类型、静态规则或构建检查中与改动相关的部分。

### L3：高风险

适用于公共契约、数据、权限、计费、多租户、并发、迁移、生产运行路径和不可逆影响。

在 L2 基础上按实际风险增加：

- 兼容性和旧数据验证；
- 权限拒绝、租户隔离和滥用路径；
- 幂等、并发、一致性、超时和恢复；
- 迁移演练、回滚和数据对账；
- 安全、性能、容量或可观测性检查；
- 独立审查和修复后复验。

不得因为任务被分类为 L3 就无条件运行全仓库、全浏览器或昂贵外部测试。根据影响半径选择最有信息量的证据。

## 三、比例化新鲜证据完成门

完成门只收取当前差异、当前风险和当前 MUST 所需的最小充分证据；不要把“通过更多命令”误当成更高可信度。

| 层级 | 宣布完成前至少具备 | 不要求 |
|---|---|---|
| L1 | 与改动直接相关的一项新鲜定向检查或等价人工观察、当前 diff 核对、未执行项披露 | 全仓库测试、浏览器矩阵或独立审查 |
| L2 | L1，加上可复现目标行为的测试、受影响接缝验证和一条核心路径烟雾检查 | 与变更无关的服务或昂贵外部调用 |
| L3 | L2，加上当前风险对应的兼容、权限、数据、并发、迁移、恢复或观测证据，以及适用的独立审查/复验 | 无边界地运行所有环境或用覆盖率数字代替风险证据 |

每个计入完成门的证据都要能关联到验收标准、命令/操作、结果、当前变更标识和失效条件。低风险任务达到 L1 即可关闭；L2/L3 不能拿 L1 的旧输出、计划或代码阅读冒充完成。任何必须检查标为 `N/A` 时，写明不适用理由和剩余风险；未执行不等于通过。

## 四、按变更资产启用门禁

只启用与当前改动直接相关的门禁：

| 变更资产 | 最低关注点 |
|---|---|
| 前端与交互 | loading、empty、error、success、重复提交、恢复、关键尺寸、键盘和可访问性 |
| API 与服务 | 输入、鉴权授权、状态码、错误结构、幂等、超时、兼容性和前后端合约 |
| 数据与迁移 | 类型约束、事务、索引、旧数据、并发、隔离、回滚和对账 |
| 异步与长任务 | 入队、终态、轮询或回调停止条件、取消、重试、超时、重复副作用 |
| 权限与计费 | 拒绝路径、租户边界、额度原子性、失败返还、重复扣减和审计 |
| 外部集成 | 请求合约、签名、限流、超时、畸形响应、降级和真实费用边界 |
| 性能与并发 | 热路径、无界操作、竞争条件、重复工作、代表性基线和回归阈值 |
| 构建与运行 | 环境变量、健康检查、日志、关键指标、发布兼容和回滚入口 |

## 五、测试与替代证据

- 优先使用能够复现需求或缺陷的现有测试层级。
- 新增核心行为时补充最接近业务边界的测试，不为覆盖数字制造无意义断言。
- 项目已有覆盖率门禁时遵循门禁；没有门禁时按风险覆盖关键分支，不强行把遗留全库提升到固定比例。
- 外部服务默认使用符合真实合约的替身、模拟服务或本地环境，未经授权不产生真实费用或外部写入。
- 无法自动化时，记录可重复的人工步骤、输入、观察结果和限制。
- 无法执行必要验证时，不得把推断或代码阅读写成“已通过”。

## 六、新鲜证据规则

验证证据至少记录：

- 关联的验收标准和影响范围；
- 实际执行的命令、操作或检查；
- 执行结果和关键原始输出；
- 对应的当前文件差异、变更标识或执行时间；
- 未覆盖区域与剩余风险。

以下变化会使旧证据失效或需要重新评估：

- 修改了被验证的生产代码或测试；
- 修改了共享接口、Schema、状态机或公共依赖；
- 改变了环境、构建配置、迁移或数据 Fixture；
- Builder 根据 Critic 发现实施的修复影响了已通过路径；
- 当前工作区与证据关联的变更标识不再一致。

未经实际执行、与当前差异不匹配、来自旧实现或无法追溯来源的证据都不是新鲜证据。

## 七、失败与收敛

验证失败时区分实现、测试假设、环境、依赖和用户已有改动，修复真实原因后重跑受影响层级。不要只增加等待时间、重试次数、放松断言或跳过测试。

完成前确认：

- 每个 MUST 都有当前实现和新鲜验证证据；
- 关键正常、失败和边界路径已按风险覆盖；
- 受最后一轮修改影响的证据已复验；
- 未执行项和外部限制已明确披露；
- 没有未解决的 P0/P1 问题。

缺少新鲜证据时只能报告“实现完成但尚未验证”或“受阻”，不得宣布交付完成。
