# v1.6.0｜发布门、进程边界与制品验证加固

## 发布范围

这是 `production-delivery-orchestrator` 的安全与交付可靠性小版本。`v1.5.0` 保留为直接回滚版本；本版本不移动任何历史 tag。

## 主要更新

- Release workflow 将 Ubuntu/Windows quality matrix 放入同一 tag workflow，只有 `quality` 成功后 `publish` 才会构建、attest 和创建 GitHub Release；默认 token 收敛为只读，仅 publish 获得必要写权限。
- whitespace 检查按 PR base/head、push before/head 和 tag fallback 比较真实差异；新增 Actions SHA、最小权限和已知凭据模式治理测试。
- forward runner 对 Windows 使用 Job Object，对 Linux 使用 subreaper + process group；timeout、持管道 orphan、关闭管道 orphan、`setsid` child/grandchild 均 fail-closed 并清理。stdout/stderr 总采集限制为 256 KiB。
- 三个评测 CLI 的 `--report-prefix` 只接受 portable filename；矩阵执行会在启动 Agent 前验证 profile suffix 后的完整文件名。
- Release ZIP 验证限制成员数、单成员/总未压缩大小和压缩比，以 128 KiB 分块读取验证 raw/canonical hash。

## 真实 CLI 行为证据边界

- 在隔离临时 fixture 中，当前完整技能 artifact SHA-256 为 `5e966c71686b1494ea421a567da80b87233b488ca8f9e902a627f3297f79be5b`。Codex CLI `0.144.5` 的三个真实案例均通过：模糊修复使前端 `failed` 成为终态、只读诊断识别轮询/终态不一致、结果选项基于仓库证据给出“手动重试（推荐）”。逐案例的脱敏可引用摘要见 [真实 CLI 矩阵证据](evidence/v1.6.0-real-cli-matrix.md)。
- Claude Code `2.1.212` 三例均在执行前因隔离环境没有登录态而退出：`Not logged in · Please run /login`。Gemini CLI `0.51.0` 三例均以 exit 41 退出，因为隔离环境未提供 `GEMINI_API_KEY`、Vertex 或 GCA 认证。两者均未修改 fixture 或技能 artifact；这不是技能行为失败的证据，也不是跨客户端行为通过的证据。
- 完整三客户端真实行为矩阵仍未完成。要复跑，必须在仓库外以短期凭证文件传入 `--agent-env-file`，且不得提交或打印其内容：Claude 使用 `ANTHROPIC_API_KEY`（建议 profile 的非交互 bare 模式；[官方 headless 文档](https://code.claude.com/docs/en/headless.md) 与 [环境变量文档](https://code.claude.com/docs/en/env-vars.md) 还定义了隔离 `CLAUDE_CONFIG_DIR`），Gemini 使用 `GEMINI_API_KEY` 或官方 Vertex/服务账号变量（[官方认证文档](https://github.com/google-gemini/gemini-cli/blob/8d951de3855750d5f8219d65ae22524b606133b6/docs/get-started/authentication.mdx)）。浏览器 `/login`、宿主 OAuth 缓存或把 `.gemini/.env` 写入 fixture 都不能替代该隔离证据门。
