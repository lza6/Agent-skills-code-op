把“看起来能用”压缩成可核验的事实

这是给下一位维护者的交接报告，不是宣传页。它区分已运行的证据、已修复的缺陷、仍受外部条件限制的真实 Agent 行为，以及下一次新增 API 或功能时应复用的工作流。

上下文改了什么工作流证据边界测验

1. 上下文与直觉

该仓库是一个 Python 标准库实现的 Agent Skill 分发、安装、离线评测和 Release 工程；它不是 Web/SaaS 应用。因此，本轮没有凭空添加前端页面、数据库、API、支付或容器。真正要保护的用户旅程是：发现 Skill → 安装到指定目录 → 读取按需契约 → 离线评测或受控真实 CLI → 双平台质量门 → 可验证的 Release。

核心判断：对这种仓库最危险的不是“缺一个酷炫框架”，而是历史事实冒充当前事实、安装器在 Windows 上不可读、浅克隆评测无恢复路径，以及把静态分数误称为真实 Agent 行为。
Stable releasev1.7.0 保持不可变；本轮工作位于 001-production-audit。
Architecture单 Skill、Python 3.11 CI、stdlib-only、GitHub Actions 双平台质量门。
Truth boundary静态 eval、安装和图谱不等于真实模型/客户端行为。

3. 下次新增 API 或功能时怎样复用

  1. 先读取项目规则、current-state、活动规格和 workflow_status，再用 Git/代码/命令确认哪些事实仍新鲜。
  2. 将需求拆成有依赖、验收、证据和回滚点的节点；只有文件不重叠且无顺序依赖时才并行。
  3. 修改前先让缺陷重现或建立等价反证；修改后按风险跑单元、接缝、安装、release 或真实客户端验证。
  4. 让独立只读 Critic 从需求完整性、逻辑、边界、代码质量、测试覆盖和实际运行六维挑战结果；修复后再复验。

具体规则在 maintenance contract；它明确禁止把历史、mock、静态 eval 或安装成功包装成当前真实运行。

4. 已有证据与验收口径

  • Spec Kit constitution、规格、计划、任务和需求追踪已落在仓库中，而不是只存在于会话。
  • 本机 CodeGraph 已真实索引 27 个代码文件、773 个节点和 1,286 条边，并查询到 InstallTransaction 及其事务方法；它提供导航证据，不替代运行正确性。
  • 已按 TDD 先运行新合同/报告测试并得到预期红灯，再实现对应文件；最终结果以 测试台账 的本轮命令为准。
  • 发布质量门要求 `publish` 依赖双平台 quality；稳定 tag 没有被移动。当前事实由 current-state 而非历史 Release 报告定义。
  • 安全分析覆盖安装目录、ZIP、外部 CLI 配置、凭证和 Release 供应链，见 威胁模型。

5. 仍然不能伪称完成的边界

真实跨 CLI 行为仍是 partial。Codex 的 hardened runner 已有 3/3 真实样本；Claude 本机 OAuth 没有可用模型；Gemini 没有隔离 API/Vertex 凭证。因此离线评测、CLI probe 和安装结果不能写成“三客户端已通过”。
  • Understand Anything 的固定版本已安装并构建，但其 `.ua` 图谱生成尝试被 Windows sandbox 的 os error 206 阻断；没有把它当成代码正确性证明。独立的本机 CodeGraph 索引已可用于导航，但同样不等于行为验证。
  • Figma、前端设计、MCP Server、Composio 等能力在本仓库没有真实输入或产品面；本轮仅如实评估其适用性，未制造无用依赖。
  • 任何新 tag、push、发布、真实模型调用或外部 OAuth 行为都需要重新获得对应环境结果，并遵循 release 过程。
